# Sample Folder Structure

Portable guide. Replace `<sample-root>/` with detected app root.

Detect root first:

1. Current directory has app `go.mod` → `.`.
2. Else `examples/golang-sample/go.mod` → `examples/golang-sample`.
3. Else ask user.

## Module

If app depends on govern locally:

```text
require github.com/haipham22/govern v0.0.0
replace github.com/haipham22/govern => ../../
```

No `go.work` unless target project uses one. Govern is optional external dependency.

## Top Level

```text
<sample-root>/
├── main.go                  # Thin app entry, calls cmd.Execute
├── cmd/                     # Cobra commands
├── internal/                # Private app code
├── pkg/                     # Small app support packages
├── .mockery.yml             # Mock generation config (optional)
├── go.mod
└── Makefile
```

## `cmd/`

Purpose: command orchestration only. Names follow project binaries.

```text
cmd/
├── root.go                  # root command, shared flags
├── serverd.go               # HTTP server (or project name)
├── grpcd.go                 # gRPC server, if present
└── workerd.go               # background worker, if present
```

Rules:

- Parse flags.
- Load config.
- Call `internal/bootstrap`.
- Defer cleanup immediately.
- Return errors; do not bury business logic here.

## `internal/bootstrap/`

Purpose: manual DI composition root. Replaces Wire.

Owns:

- logger setup
- database/cache setup
- repository construction
- usecase construction
- handler/controller construction
- server construction
- cleanup aggregation

Construction order:

```text
config → logger → database/cache → repositories → usecases → handlers/controllers → server
```

Resource constructors that hold handles return:

```go
(T, func(), error)
```

Cleanup runs reverse construction order.

## `internal/domain/`

Purpose: pure business entities.

Contains:

- `User`, `Product`, value objects
- domain validation that needs no infra
- domain-specific sentinel errors

Must not import:

- Echo
- GORM
- Redis
- Zap
- Viper
- handler/usecase/repository packages

Do not put repository interfaces here. Interfaces belong to consumer layer.

## `internal/schemas/`

Purpose: HTTP boundary DTOs.

Contains:

- request structs
- response structs
- JSON tags
- boundary mapping helpers when needed

Rules:

- Handler binds into schemas.
- Usecase should receive usecase/domain input, not Echo context.
- Do not expose `orm` structs as API responses.

## `internal/handler/`

Purpose: delivery adapters.

```text
handler/
├── rest/                    # Echo HTTP routes, middleware, controllers
├── grpc/                    # gRPC delivery
├── job/                     # cron delivery
└── message/                 # asynq/message delivery
```

Rules:

- Bind/validate request.
- Convert to usecase input.
- Call usecase.
- Map result/error to protocol response.
- No DB calls.
- No business policy.

## `internal/usecase/`

Purpose: application business logic and consumed interfaces.

```text
usecase/
├── auth/
├── product/
└── user/
```

Rules:

- Define repository interfaces here when usecase consumes storage.
- All I/O methods take `context.Context` first.
- Coordinate business flows.
- Own multi-step transaction decisions at workflow boundary.
- Do not import repository implementations.
- Do not know Echo/GORM details.

## `internal/repository/`

Purpose: infrastructure adapters.

```text
repository/
├── postgres/                # GORM-backed repositories
├── redis/                   # Redis-backed repositories
└── user/                    # user storage adapter/interface compatibility
```

Rules:

- Implement usecase interfaces.
- Use `db.WithContext(ctx)` for GORM.
- Map `orm` ↔ `domain` / usecase objects.
- Wrap infra errors with context.
- No business policy.

## `internal/orm/`

Purpose: persistence models only.

Rules:

- GORM tags live here.
- Repository may use these.
- Handler/usecase should not return these directly.

## `internal/validator/`

Purpose: `go-playground/validator` wrapper.

Rules:

- Keep validation errors client-safe.
- Use JSON tag names for field names.
- Handler or usecase can call validation depending on boundary.

## `internal/errors/`

Purpose: app custom errors (package name follows project; e.g. `apperrors`).

If app migrated off `govern/errors`, keep app-specific envelope. Do not reintroduce `govern/errors` into request flow unless project decision changes.

## `internal/mocks/`

Purpose: generated test mocks.

Rules:

- Generated by Mockery v3 via `.mockery.yml`.
- Do not edit `mock_*.go` manually.
- Regenerate after interface changes.

## `pkg/`

Small app support packages:

```text
pkg/
├── config/                  # sample env/config structs
├── postgres/                # sample DB connection helper
└── utils/                   # password/string helpers
```

Keep `pkg/` boring. Do not move business logic here.

## Import Direction

```text
handler → usecase → domain ← repository
bootstrap → all layers
```

Forbidden:

- `domain → usecase/repository/handler`
- `usecase → handler/repository implementation`
- `handler → repository/orm`
- `repository → handler`
